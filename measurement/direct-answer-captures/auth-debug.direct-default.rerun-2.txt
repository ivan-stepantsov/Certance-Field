The likely root cause is an expiry-boundary bug, not a random session failure. “After exactly one hour” usually means the middleware is rejecting the session at the precise `exp` cutoff because the comparison is wrong for the token format or timestamp rounding, typically an off-by-one check such as treating `now === exp` as already invalid when the rest of the stack still considers it valid.

The smallest plausible fix is to change the expiry check so it only rejects once the current time is strictly past the expiry boundary, and make sure both sides use the same unit. In practice that usually means fixing a comparison like `now >= exp` or `age >= 3600` to the correct boundary for your token contract, often `now > exp`, or changing a mismatched `<`/`<=` pair in the validation branch. If the code mixes milliseconds and seconds, normalize that first, because a boundary bug plus unit mismatch is the other common exact-one-hour failure.
